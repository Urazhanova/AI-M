# Правила оформления лекций и работы с материалами

Единый источник правды для куратора, агента и любого, кто вносит материалы в трекер.
Если в коде или в `CLAUDE.md` обнаружено расхождение с этим документом — приоритет у этого документа.

---

## 1. Где что лежит

```
AI-M/
├── inbox/                    # DROP ZONE: куратор кладёт сюда новые транскрипты вручную (если не через UI)
├── raw/                      # АРХИВ ИСТОЧНИКОВ: все транскрипты в исходном виде, неизменяемы
├── course/
│   ├── sequence.md           # граф связей чанков (узлы и leads_to)
│   └── [chunk-id]/
│       ├── content.md        # учебный материал (Markdown)
│       └── meta.md           # метаданные по схеме раздела 5
├── students/[name]/
│   ├── profile.md            # портрет студента
│   ├── my-path.md            # индивидуальный путь (таблица статусов)
│   └── progress.md           # лог прохождения (append-only)
├── docs/
│   └── lecture-pipeline.md   # этот файл
├── index.md                  # авто-обновляемый общий индекс
└── mindset-map.md            # карта mindset-сдвигов всех студентов
```

**Жёсткие инварианты:**
- Файлы лекций (`content.md`, `meta.md`) лежат **только** внутри `course/[chunk-id]/`. Никаких `.md` в корне `course/`.
- Сырые транскрипты лежат **только** в `raw/`. Никогда — в `course/`, `inbox/` (после обработки), или в корне репо.
- `inbox/` — временная папка. После обработки файл переезжает в `raw/`. Не должен оставаться надолго.

## 2. Конвенция именования

### chunkId
Формат: `s{N}-w{N}-{type}[-suffix]`

| Поле | Значения |
|---|---|
| `s{N}` | номер спринта (`s3`) |
| `w{N}` | номер недели (`w1`); для вводной встречи всего спринта — `intro` без `w` |
| `type` | `intro` \| `lecture` \| `practical` \| `workshop` |
| `-suffix` | `-a`, `-b`, `-c` … когда одна сессия даёт несколько юнитов; либо смысловой суффикс `-setup`, `-recap` |

**Примеры:**
- `s3-intro` — вводная встреча спринта
- `s3-w1-lecture` — единственная лекция первой недели
- `s3-w1-lecture-a`, `s3-w1-lecture-b`, `s3-w1-lecture-c` — три юнита из одной двухчасовой лекции
- `s3-w2-workshop-setup` — отдельный сетап-воркшоп

**Уникальность:** `chunkId` уникален в пределах спринта. Один `chunkId` = одна папка в `course/`.

### Имена файлов транскриптов в `raw/`
Формат: `[chunk-id]-transcript-[YYYY-MM-DD].md`
- При склейке нескольких файлов одной сессии: суффикс `-merged`.
- Никаких префиксов из календаря (`пн`, `вт`, `ср`), никаких пробелов в имени, никаких `(1)`, `(2)`.
- Только латиница, цифры, дефисы.

**Примеры:**
- `raw/s3-w1-lecture-transcript-2026-05-05.md`
- `raw/s3-w1-practical-merged-transcript-2026-05-05.md`

## 3. Правила разбивки лекции на юниты

Юнит — самостоятельная смысловая единица, которую можно изучить отдельно за 20–40 минут.

| Длительность исходника | Сколько юнитов |
|---|---|
| < 30 минут | 1 юнит, без суффикса |
| 30–60 минут | 1–2 юнита |
| 60–120 минут | **2–4 юнита**, с суффиксами `-a`, `-b`, `-c` |
| > 120 минут | 3–6 юнитов |

**Что НЕ юнит:**
- Слайд из лекции
- Один пример без контекста
- Пятиминутное отступление

**Что юнит:**
- Раздел лекции с собственной идеей и собственным DoD (можно проверить отдельно)
- Практика с конкретным артефактом
- Воркшоп с самостоятельным результатом

## 4. Workflow куратора

### Сценарий 1: загрузка через UI (рекомендуемый)

1. Куратор открывает `/curator/upload`
2. Выбирает `.txt` или `.md` файл с транскриптом
3. Опционально вводит ID-подсказку (например `s3-w2-lecture`)
4. Жмёт «Обработать»
5. Пайплайн:
   - **Splits** транскрипт на 1–6 юнитов (Gemini)
   - Для каждого юнита создаёт `course/[chunkId]/content.md` и `meta.md` (полная схема — раздел 5)
   - Добавляет узлы в `sequence.md`
   - Добавляет строки в `students/*/my-path.md` со статусом `⬜ не начат`
6. Куратор проверяет результат на дашборде, при необходимости правит вручную

### Сценарий 2: ручная загрузка (если UI недоступен)

1. Кладёшь транскрипт в `inbox/` под именем `[chunk-id]-transcript-[YYYY-MM-DD].md`
2. Сообщаешь агенту: «обработай новые транскрипты»
3. Агент проходит по `inbox/`, для каждого:
   - Читает транскрипт
   - Решает, к какому `chunkId` относится (или это новый чанк)
   - Создаёт/обновляет `course/[chunkId]/content.md` и `meta.md`
   - Перемещает файл в `raw/` с правильным именем
4. Агент обновляет `sequence.md`, `index.md`, `students/*/my-path.md`

### Что нельзя

- Класть `.md` в корень `course/`. Любой такой файл — ошибка, мигрировать в `raw/`.
- Создавать чанк без `meta.md`. `meta.md` обязателен по схеме раздела 5.
- Редактировать старые записи в `progress.md` (только append).
- Удалять файлы из `raw/` — это первичный архив.

## 5. Схема `meta.md`

Каждый `meta.md` обязан иметь все 5 секций в этом порядке:

```markdown
# meta: [chunk-id]

\`\`\`yaml
id: [chunk-id]
title: "Название юнита"
session_type: lecture | practical | workshop | intro
date: YYYY-MM-DD
duration_min: <число>
sprint: sN
week: wN | intro
position: <число>
speaker: "Имя или пусто"
\`\`\`

## Learning Outcomes (с таксономией Блума)

### Knowledge — что студент должен помнить и понимать
| Знание | Уровень Блума |
|---|---|
| ... | 1-2 |
(минимум 3 строки)

### Skills — что студент должен уметь делать
| Навык | Уровень Блума |
|---|---|
| ... | 3-6 |
(минимум 3 строки)

### Attitudes — какие установки и mindset-сдвиги формирует юнит
| Установка | Уровень Блума |
|---|---|
| ... | 4-5 |
(минимум 2 строки)

## Проверка усвоения

### Знание (Knowledge)
- ...
### Понимание (Comprehension)
- ...
### Применение (Application)
- ...
### Анализ (Analysis)
- ...
### Синтез (Synthesis)
- ...
### Оценка (Evaluation)
- ...

## Definition of Done

### Минимальный DoD (Bloom 1-3)
- <конкретный артефакт или действие>

### Полный DoD (Bloom 4-6)
- <артефакт уровня анализа/синтеза/оценки>

## Граф связей

\`\`\`yaml

\`\`\`
```

**Таксономия Блума:**
1 — Помнит, 2 — Понимает, 3 — Применяет, 4 — Анализирует, 5 — Оценивает, 6 — Создаёт.

## 6. Схема `content.md`

```markdown
# [Название юнита]

**Дата:** YYYY-MM-DD
**Формат:** [lecture | practical | workshop | intro]
**Спикер:** [Имя или «не указан»]

---

## [Раздел 1]
...
## [Раздел 2]
...
```

Правила:
- Минимум 400 слов на юнит. Меньше — недостаточно для проверки усвоения.
- Заголовки H2 для логических разделов; H3 для подразделов.
- Без слов-паразитов и артефактов транскрипта (повторов, «эээ», тайм-кодов).
- Только Markdown — без HTML.

## 7. `sequence.md` — единый источник правды по графу

Узлы графа в `sequence.md` должны **всегда** соответствовать `meta.md` соответствующего чанка. Если расхождение — `meta.md` приоритетнее, `sequence.md` приводится к нему.

При добавлении нового юнита через пайплайн `sequence.md` обновляется автоматически. Ручные правки только для:
- Уточнения `leads_to` и `related` (пайплайн ставит пустые массивы)
- Обновления визуального ASCII-графа в начале файла

---

## 8. `my-path.md` студента

Таблица «Текущий статус» содержит все чанки спринта со статусами:

| Статус | Когда |
|---|---|
| `⬜ не начат` | юнит существует, студент не приступал |
| `🟡 в работе` | начал, но DoD не закрыт |
| `✅ пройден` | минимальный или полный DoD закрыт |

При создании нового юнита пайплайн добавляет строку со статусом `⬜ не начат` для **каждого** студента в `students/`.

## 9. Ветка и коммиты

- Единственная ветка: **`front`** (default branch на GitHub)
- Все автокоммиты пайплайна идут в `front`
- Сообщение коммита от пайплайна: `[AI] Добавлен юнит {chunkId}: {title}` или `[AI] Метаданные {chunkId}` или `[AI] my-path.md {student}: добавлены новые юниты`
- Ручные коммиты студентов: `{name}: {chunkId} — {DoD статус}`

## 10. Чек-лист «лекция в порядке»

Прежде чем считать лекцию готовой:

- [ ] Транскрипт лежит в `raw/`, имя по конвенции
- [ ] Все юниты лекции есть в `course/[chunkId]/` с `content.md` и `meta.md`
- [ ] Каждый `meta.md` соответствует схеме раздела 5 (5 секций)
- [ ] `sequence.md` содержит узел для каждого `chunkId` и описание совпадает с `meta.md`
- [ ] У всех 3 студентов в `my-path.md` есть строка для каждого юнита
- [ ] Дашборд `/curator` показывает корректное число чанков
- [ ] Дашборд `/student/[name]` показывает все юниты в пути
