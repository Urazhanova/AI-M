1. Назначение системы
AI Mindset Learning Tracker — веб-приложение для управления образовательными программами, отслеживания прогресса студентов и визуализации mindset-сдвигов. Система заменяет ручную работу с Markdown-файлами на структурированный интерфейс, сохраняя совместимость с Git-репозиторием.
Основные задачи:
	•	Единая точка входа для кураторов и студентов
	•	Визуализация прогресса по чанкам, уровням Блума и mindset-статусам
	•	Управление несколькими образовательными программами и когортами студентов
	•	Проведение учебных сессий с записью прогресса через интерфейс
	•	Автоматическая синхронизация данных с GitHub


2. Пользователи и роли

3. Экраны и функциональность
3.1 Аутентификация
Вход через GitHub OAuth. Позволяет использовать существующие аккаунты и автоматически связывать пользователей с данными в репозитории.
	•	Страница входа: кнопка «Войти через GitHub»
	•	При первом входе: куратор вручную назначает роль новому пользователю
	•	Выход доступен в любой момент из шапки

3.2 Главный дашборд (куратор)
Стартовый экран с сводной информацией по всем активным программам.

Блок
Содержание
Метрики сверху
Количество студентов, чанков, сессий за неделю, полных DoD
Карточки студентов
Прогресс-бар, количество чанков, последний DoD, кнопка перехода к студенту
Таблица чанков
Список всех чанков программы с уровнями Блума и статусами DoD по каждому студенту
Быстрые действия
Кнопки «Начать сессию», «Добавить программу», «Добавить студента»

3.3 Карточка студента
Детальный профиль студента. Доступен куратору и самому студенту (только свои данные).
	•	Имя, программа, дата начала
	•	Прогресс по каждому чанку: статус DoD, достигнутые уровни Блума
	•	Лог прогресса: хронологический список сессий с рефлексиями
	•	Mindset-статусы: какие сдвиги seeded / emerging / internalized
	•	Кнопка «Начать сессию» — переходит к экрану сессии

3.4 Учебная сессия
Структурированный режим работы с чанком. Куратор ведёт сессию, фиксирует ответы студента и результаты.

Шаг
Действие
Результат
1. Выбор чанка
Куратор выбирает чанк из графа последовательности
Система показывает content.md в адаптированном виде
2. Подача материала
Отображается адаптированный текст под профиль студента
Студент читает, задаёт вопросы
3. Проверка
Вопросы из meta.md по уровням Блума, по одному
Куратор отмечает достигнутые уровни
4. Закрытие
Выбор DoD, рефлексия, автокоммит в GitHub
Данные сохраняются и синхронизируются

3.5 Mindset map
Сводная таблица mindset-сдвигов по всем студентам. Обновляется автоматически после каждой сессии.
	•	Строки: mindset-сдвиги («от → к»)
	•	Колонки: студенты
	•	Ячейки: цветной статус (seeded / emerging / internalized / —)
	•	Клик по ячейке: открывается заметка с контекстом сдвига

3.6 Управление программами
Раздел для куратора. Позволяет создавать новые образовательные программы и управлять существующими.
	•	Список программ: название, статус (активна / черновик / завершена), количество студентов и чанков
	•	Создание программы: название, описание, структура спринтов и чанков
	•	Редактор чанков: загрузка content.md и meta.md, редактирование через веб-форму
	•	Управление студентами: добавление, удаление, назначение программы

3.7 Личный кабинет студента
То, что видит студент при входе. Только его данные.
	•	Мой прогресс: список чанков, статусы, следующий шаг
	•	Мои mindset-сдвиги: только его строка из таблицы
	•	История сессий: даты, рефлексии, уровни Блума
	•	Кнопка «Запросить сессию» — уведомление куратору

4. Модель данных
Система использует гибридный подход: Markdown-файлы в GitHub как источник истины, БД как кэш для быстрого отображения. При каждой синхронизации БД обновляется из репозитория.

id, program_id, title, type, bloom_levels

Правило: Папка inbox/ не попадает в веб-интерфейс и не коммитится — она обрабатывается агентом Claude отдельно.

5. Интеграция с GitHub
GitHub выполняет роль единого хранилища и системы контроля версий. Веб-приложение читает и пишет файлы через GitHub API, сохраняя полную совместимость с существующей структурой папок.

5.1 Операции чтения
	•	При загрузке: получить список чанков из course/, профили студентов из students/
	•	При открытии карточки: прочитать progress.md, my-path.md, profile.md
	•	При старте сессии: прочитать content.md и meta.md выбранного чанка
	•	При открытии Mindset map: прочитать mindset-map.md

5.2 Операции записи
После закрытия сессии система автоматически формирует коммит:
	•	Дописывает новую запись в students/[name]/progress.md (append-only)
	•	Обновляет students/[name]/my-path.md
	•	Обновляет mindset-map.md (если есть attitude-сдвиг)
	•	Коммит: автор — куратор, шаблон: [name]: [chunk-id] — [статус DoD]
	•	Перед записью: git pull --rebase, чтобы не перезаписать изменения других

7. Этапы разработки
Разработка разбита на три этапа. Каждый этап — рабочий, задеплоенный продукт.

Этап 1 — MVP (2–3 недели)
Цель: заменить работу с Markdown-файлами на веб-интерфейс с базовыми функциями.
	•	Вход через GitHub OAuth
	•	Главный дашборд: карточки студентов, список чанков
	•	Карточка студента: прогресс, DoD, Блум
	•	Mindset map: статичная таблица
	•	Чтение из GitHub, запись через API после сессии

Этап 2 — Сессии и множественные программы (2–3 недели)
Цель: полноценное проведение учебных сессий через интерфейс.
	•	Экран учебной сессии: адаптированный материал + вопросы по Блуму
	•	Форма закрытия: DoD, рефлексия, автокоммит
	•	Управление программами и чанками
	•	Личный кабинет студента

Этап 3 — Аналитика и масштаб (1–2 месяца)
Цель: полноценная платформа для нескольких курсов.
	•	Граф последовательности чанков (визуализация)
	•	Аналитика: когорты, динамика mindset-сдвигов
	•	Роль наблюдателя
	•	Экспорт отчётов в PDF

8. Нефункциональные требования

Требование
Описание
Скорость
Страницы открываются менее чем за 2 секунды. Данные из Supabase кэшируются, GitHub API вызывается только при записи.
Безопасность
Студент видит только свои данные. GitHub-токен хранится в переменных окружения на сервере, не в коде.
Совместимость
Все данные остаются в Markdown и совместимы с Claude-агентом. Веб — дополнение, не замена.
Надёжность
Если GitHub API недоступен, система работает в режиме чтения из Supabase. Запись ставится в очередь.

9. Открытые вопросы
Следующие решения требуют обсуждения до начала разработки:

	•	Нужен ли студентам самостоятельный вход, или они работают только через куратора на одном устройстве?
	•	Как обрабатывать транскрипты (inbox/)? Загрузка через веб-интерфейс или по-прежнему через агента?
	•	Нужна ли локализация (русский/казахский) с первого этапа?
	•	Кто разрабатывает: штатный разработчик, фрилансер, или low-code (например, Bubble, Softr)?
	•	Нужна ли интеграция с Telegram для уведомлений о новых сессиях?
